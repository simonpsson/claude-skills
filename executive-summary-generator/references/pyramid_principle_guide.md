# Pyramid Principle Guide

## The core idea

Barbara Minto's Pyramid Principle: lead with the answer, then provide supporting arguments, then evidence. This is the inverse of how most analysts naturally write — they present the analysis first and the conclusion last.

The pyramid structure:

```
Governing thought (the answer / recommendation)
├── Key line 1 (first supporting argument)
│   ├── Evidence 1a
│   └── Evidence 1b
├── Key line 2 (second supporting argument)
│   ├── Evidence 2a
│   └── Evidence 2b
└── Key line 3 (third supporting argument)
    ├── Evidence 3a
    └── Evidence 3b
```

---

## The governing thought

The governing thought is a single sentence that answers the question the document addresses. It must be:

- **Specific:** "We should invest in paid acquisition in EMEA" not "We should think about our acquisition strategy"
- **Declarative:** A statement, not a question or a description
- **Defensible:** Something you can support with the key lines below

Test: if a senior executive read only the governing thought, would they know what you want them to do or believe?

---

## Key lines

Key lines are the top-level supporting arguments. They should be:

- **Mutually exclusive:** No two key lines make the same point
- **Collectively exhaustive:** Together, they fully support the governing thought
- **Parallel in structure:** Either all diagnoses, all recommendations, or all reasons — not a mix

**Three key lines is the ideal.** Two feels thin; four requires trade-offs in depth.

---

## MECE principle

MECE = Mutually Exclusive, Collectively Exhaustive.

- **Mutually exclusive:** No overlap between groups (a user cannot be in two segments)
- **Collectively exhaustive:** All cases are covered (no user falls outside all segments)

Apply MECE when structuring key lines, segmentation, and any list of options.

**Quick test:** Can you name a case that belongs to two categories? (Not ME.) Can you name a case that belongs to no category? (Not CE.)

---

## When the pyramid works and when it doesn't

**Works well for:**
- Executive memos
- Slide presentations
- Email recommendations
- One-page summaries

**Works less well for:**
- Building shared context before a controversial conclusion
- Exploratory discussions where the audience needs to co-create the answer
- Findings that require extensive setup before they will be believed

In those cases, use SCR (Situation, Complication, Resolution) instead — it defers the governing thought until the audience has the context to accept it.

---

## Common mistakes

**Stating the topic instead of the answer:** "This memo covers Q3 revenue performance" is a topic, not a governing thought. Replace with: "Q3 revenue was 12% below target because of EMEA underperformance."

**Evidence at the key line level:** Don't put data points as key lines. Key lines are arguments; evidence lives below them.

**Too many levels:** Stop at 3 levels (governing thought → key lines → evidence). More creates cognitive overload.

**Unparallel key lines:** "Conversion is declining" (diagnosis) + "Improve onboarding" (recommendation) + "Mobile users are the problem" (cause) — three different types of statements. Make them all diagnoses, all causes, or all recommendations.
